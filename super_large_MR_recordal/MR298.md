### 三、申请例外原因：

本 PR 实现基于 GBP 的 oGRAC 集群 RTO 加速能力，改动围绕 RT 实时分析、GBP/GBPS 页面镜像旁路、undo 恢复优化三部分展开。该能力需要同时打通集群 redo 实时分析、内核 GBP 写页与恢复拉页、GBPS 独立组件、CMS/appctl 生命周期管理、DTC partial recovery、checkpoint、redo recovery、buffer、事务恢复等多条链路，属于完整特性的整体回合。

本需求已按功能域进行拆分设计和提交组织，主要包括 RT 分析能力、GBPS 服务端组件、内核 GBP 路径、CMS/appctl 生命周期管理、恢复 fallback 路径以及事务和 planner 防护等部分。但这些部分之间存在强依赖关系：RT 分析结果用于构造部分恢复 required page 集合；GBP/GBPS 需要在运行期保存脏页镜像，并在恢复期按照 required page 元数据拉取页面；恢复链路还需要对拉页失败、窗口不安全、skipped redo page 校验失败等场景 fallback 到 redo 恢复。因此继续拆分为多个独立可合入 MR 会导致中间状态不可独立验证，且容易引入不可用或不可回退的恢复路径。

本 PR 的超大行数主要来自新增 GBPS 独立组件、内核 GBP 模块、DTC GBP RT 分析模块，以及对既有恢复链路的大范围接入改造。其中新增文件包含 GBPS 配置、协议、状态管理、管理命令、RT 分析、GBP 内核复制模块等；修改文件集中在 `pkg/src/cluster`、`pkg/src/gbp`、`pkg/src/kernel/replication`、`pkg/src/kernel/persist`、`pkg/src/kernel/buffer`、`pkg/src/cms` 和部署脚本等目录。

PR 描述中已附开发自验截图，自验结果显示 RTO 为 8.852s，小于 10s 目标。实现中同时补充了异常路径保护，包括 RT gap/unsafe redo/窗口不可用时主动回退、GBP 拉页失败或校验不满足时 fallback 到 redo、GBPS uninstall 强制清理残留进程、undo free list 按 segment 重建以降低恢复阶段锁开销，以及 planner 中 numeric text index 访问防护。
