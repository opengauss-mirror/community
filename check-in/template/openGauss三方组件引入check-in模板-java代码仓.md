| 检查项                   | 详细说明                                                     | 是否满足 |
| ------------------------ | ------------------------------------------------------------ | -------- |
| 必要性                   | 引入必要性说明（引入会有什么优势，不引入会有什么影响）       |          |
| 归一化                   | 同一仓库同一软件原则上只有一个版本优选，对同一软件多版本的要进行精简、优化归一，避免造成多余引入，浪费人力，以及后期大量的漏洞管理 |          |
| 来源可靠                 | 不能使用来源不明的，不能随意从网上拷贝一份代码使用；是否是业界广泛认可的主流社区或主流项目的开源软件，在质量、安全、可维护性等方面有较好的保障 |          |
| 版本检查                 | 1、版本名称必须和官网/社区保持一致；2、待引入版本是否为近期稳定的官方发布版本（Release版本），非正式版本未经过全面测试，不建议入库；3、对于没有正式发布版本的、被动依赖导致只能引入非正式发布版本的，需要在社区安全sig进行评审；4、不能以master分支作为版本引入 |          |
| 网络安全风险检查         | 1、待引入版本是否有高危漏洞，漏洞查询可参考NVD漏洞库；3、优选提供官方漏洞、补丁、缺陷发布渠道的三方组件；4、通过漏洞扫描 |          |
| 版本年龄（官方发布时间） | 1、优选软件最新发布的稳定版本；2、引入版本官网发布时间，是否还处于生命周期，引入不在生命周期的版本需要在安全sig组澄清必要性并评审通过 |          |
| 官网必填                 | 1、支持使用maven、mvnrepository拉取依赖，但社区构建的版本需要使用业界公开的仓库（如maven.org、阿里云、华为云等）；2、支持使用源码引入，使用源码引入的要求与C/C++三方组件引入要求保持一致（[openGauss三方组件引入check-in模板](https://gitcode.com/opengauss/community/blob/master/check-in/template/openGauss%E4%B8%89%E6%96%B9%E7%BB%84%E4%BB%B6%E5%BC%95%E5%85%A5check-in%E6%A8%A1%E6%9D%BF.md)） |          |
| license/copyright检查    | 1、待引入软件是否有license；2、对待引入软件的具有不友好的license不建议引入：LGPL2.0，LGPL3.0，GPL2.0，GPL3.0，AGPL3.0等；3、引入软件的license和copyright需要通过门禁检查 |          |
