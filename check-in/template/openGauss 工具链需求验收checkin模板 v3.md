##  RESTful API设计规范

  RESTful API设计规范列表，涵盖了资源命名、HTTP方法使用、状态码、版本管理、过滤分页排序、错误处理、安全等方面 。

### 1. 资源命名与URI设计

- **使用名词而非动词**：URI应该表示资源，而不是操作。使用名词的复数形式。
  - 正确：`/users`、`/orders`
  - 错误：`/getUsers`、`/createOrder`
- **资源使用复数形式**：统一使用复数名词表示资源集合。
  - 例如：`/users` 表示用户集合，`/users/123` 表示ID为123的用户。
- **层级关系表示**：使用嵌套资源表示关联关系。
  - 例如：`/users/123/orders` 表示用户123的所有订单。
- **避免深度嵌套**：嵌套层级不宜过深，建议不超过两级。过深可考虑使用查询参数。
  - 例如：`/users/123/orders/456` 可接受，但 `/users/123/orders/456/products/789` 则太深，可改为 `/orders/456?user=123` 或使用其他方式。
- **使用连字符`-`提高可读性**：URI中尽量使用连字符`-`而不是下划线`_`，且不区分大小写（建议使用小写）。
  - 例如：`/api/v1/user-profiles`
- **不包含文件扩展名**：URI中不应包含文件扩展名（如`.json`、`.xml`），而是通过Content-Type头指定媒体类型。
  - 正确：`/api/v1/users` 并设置 `Content-Type: application/json`
  - 错误：`/api/v1/users.json`

### 2. HTTP方法使用

RESTful API使用HTTP方法对资源进行操作，常用的方法有：

- **GET**：获取资源（安全且幂等）
- **POST**：创建资源（非幂等）
- **PUT**：完整更新资源（幂等）或创建资源（当资源ID由客户端提供时）
- **PATCH**：部分更新资源（幂等性取决于实现，通常视为非幂等，但应设计为幂等）
- **DELETE**：删除资源（幂等）

**注意**：不要使用GET方法进行修改操作，因为GET是安全和幂等的。

### 3. 查询参数使用

对于集合资源，可以使用查询参数进行过滤、分页和排序。

- **过滤**：使用查询参数来过滤资源集合。
  - 例如：`/users?role=admin&status=active`
- **分页**：使用`page`和`size`或`limit`和`offset`等参数进行分页。
  - 例如：`/users?page=1&size=20`
  - 常用格式：`page`表示页码，`size`表示每页记录数；或者`offset`表示偏移量，`limit`表示限制数量。
- **排序**：使用`sort`参数指定排序字段和顺序。
  - 例如：`/users?sort=createdAt,desc` 或 `/users?sort=age:asc,name:desc`

### 4. 状态码使用

使用合适的HTTP状态码来表示请求结果。

- **2xx**：成功
  - 200 OK：请求成功，通常用于GET、PUT、PATCH
  - 201 Created：资源创建成功，常用于POST
  - 204 No Content：请求成功，但无返回内容，常用于DELETE
- **4xx**：客户端错误
  - 400 Bad Request：请求参数错误
  - 401 Unauthorized：未认证
  - 403 Forbidden：无权限
  - 404 Not Found：资源不存在
  - 405 Method Not Allowed：不允许的HTTP方法
  - 409 Conflict：资源状态冲突，例如重复创建
  - 422 Unprocessable Entity：请求格式正确，但语义错误（如验证失败）
- **5xx**：服务器错误
  - 500 Internal Server Error：服务器内部错误

### 5. 版本管理

API版本化，避免破坏性变更影响现有客户端。

- **URL路径版本化**：将版本号放在URL路径中。
  - 例如：`/api/v1/users`
- **请求头版本化**：使用自定义请求头指定版本，如`Accept: application/vnd.example.v1+json`。

### 6. 请求和响应格式

- **请求体**：使用JSON格式传递数据，并设置`Content-Type: application/json`。

- **响应体**：同样使用JSON格式，设置`Content-Type: application/json`。

- **统一响应格式**：对于响应，建议采用统一的封装格式，例如：

  json

  ```
  {
    "code": 200,        // 业务状态码，可选，也可直接使用HTTP状态码
    "message": "success", // 提示信息
    "data": { ... }     // 返回的数据
  }
  ```

  

  错误时：

  json

  ```
  {
    "code": 40001,
    "message": "Invalid parameters",
    "details": [ ... ]  // 可选，错误详情
  }
  ```

  

### 7. 错误处理

- **统一错误格式**：如上所述，错误响应应该包含错误码、错误信息和可选的详细信息。
- **使用HTTP状态码**：首先使用合适的HTTP状态码，然后在响应体中提供更详细的错误信息。

### 8. 安全

- **认证**：使用标准的认证机制，如OAuth2.0、JWT等。
- **授权**：对敏感资源进行权限控制。
- **HTTPS**：生产环境必须使用HTTPS。
- **敏感信息**：不要在URL中传递敏感信息（如密码、令牌），应使用请求头或请求体。

### 9. 其他

- **HATEOAS**：在响应中提供相关资源的链接，使客户端能够动态发现资源。例如：

  json

  ```
  {
    "id": 123,
    "name": "John",
    "links": [
      { "rel": "self", "href": "/api/v1/users/123" },
      { "rel": "orders", "href": "/api/v1/users/123/orders" }
    ]
  }
  ```

  

- **缓存**：合理使用HTTP缓存机制，如ETag、Last-Modified头。

- **限流**：对API进行限流，并在响应头中返回限流信息（如X-RateLimit-Limit, X-RateLimit-Remaining）。

- **文档**：提供清晰、准确的API文档，可以使用OpenAPI（Swagger）等工具。

### 10. 不符合RESTful规范的设计

- 使用动词在URI中，如`/getUser`、`/createOrder`
- 使用错误的HTTP方法，如用GET来更新资源
- 返回格式不统一，有时返回JSON，有时返回XML，或者同一接口返回不同的结构
- 滥用状态码，比如不管什么错误都返回200，然后在body里用自定义代码表示错误
